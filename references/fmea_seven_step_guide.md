# FMEA 七步法详细指南

> 基于 AIAG & VDA FMEA 手册（2019 版）整理，覆盖 DFMEA 与 PFMEA 每一步的目的、关键活动、输出物、常见错误及 IATF 16949 关联要求。

---

## 七步法总览

| 步骤 | 名称 | DFMEA 工具 | PFMEA 工具 |
|---|---|---|---|
| 1 | 规划与准备 | 5T 法、项目计划 | 5T 法、过程范围确定 |
| 2 | 结构分析 | 结构树、方块图（边界图） | 过程流程图、结构树、4M 工作要素 |
| 3 | 功能分析 | 功能树、参数图（P-图） | 过程功能分析、参数图 |
| 4 | 失效分析 | 失效链 FE→FM→FC | 失效链 FE→FM→FC（4M 分类） |
| 5 | 风险分析 | S/O/D 评级 + AP 优先级 | S/O/D 评级 + AP 优先级 |
| 6 | 优化 | 措施清单、责任人、状态 | 措施清单、责任人、状态 |
| 7 | 结果文件化 | FMEA 表格 + 报告 | FMEA 表格 + 报告 |

---

## 步骤一：规划与准备

### 目的

明确 FMEA 分析的范围、目标、团队和时间安排，确保分析方向与项目需求一致。

### DFMEA 关键活动

1. **5T 明确**：
   - **Intent（目的）**：分析是为了设计优化？满足客户要求？IATF 体系审核？
   - **Timing（时间）**：在概念设计阶段、详细设计阶段、还是设计评审阶段？
   - **Team（团队）**：设计主管（技术主导）、跨职能团队（系统/安全/工艺/质量）、FMEA 推进者
   - **Task（任务）**：分析覆盖哪些系统层级、哪些接口
   - **Tools（工具）**：使用什么 FMEA 软件或表格模板

2. **项目边界确定**：
   - 明确分析的系统层级（整车/系统/子系统/组件/零件）
   - 确定设计责任（自有设计 / 供应商设计 / 联合设计）
   - 识别接口（机械接口、电气接口、软件接口、信息接口）

3. **基准 FMEA 引用**：
   - 基础 FMEA（Foundation FMEA）：公司积累的通用 FMEA
   - 家族 FMEA（Family FMEA）：相似产品族的 FMEA

4. **工程团队协作接口**：
   - 系统团队：提供系统架构
   - 安全团队：提供安全目标和危险分析
   - 组件团队：提供组件级设计信息

### PFMEA 关键活动

1. **工艺边界确定**：
   - 分析哪条生产线、哪些工序段
   - 是否包含供应商过程？是否包含客户过程？

2. **过程流程图（PFD）确认**：
   - 列出所有工序步骤（包括运输、储存、检验等辅助工序）
   - 明确每一步的输入/输出

3. **基础 PFMEA 引用**：
   - 类似产品的 PFMEA
   - 工艺家族 PFMEA

4. **表头填写**：
   - 公司名称、制造地址、顾客名称
   - 年型/项目、项目名称
   - PFMEA 开始日期、修订日期
   - 跨职能团队名单
   - 顾客特定要求（CSR）

### 输出物

- DFMEA/PFMEA 项目立项单
- 5T 文档
- 团队成员名单与职责
- 项目计划（含里程碑）

### 常见错误

- ❌ 范围过大（如"分析整车所有失效"）—— 应聚焦关键系统/工序
- ❌ 团队成员单一（只有设计人员）—— 应跨职能（设计/工艺/质量/生产）
- ❌ 未引用基础 FMEA，重复造轮子
- ❌ 时间安排过晚（量产后才做）—— 应在 APQP 设计阶段启动

### IATF 16949 关联

- 条款 8.3.3.3（产品和过程的设计输入）
- 条款 8.5.1.1（控制计划）

## 步骤二：结构分析

### 目的

将分析对象分解为可视化的结构，识别系统要素及其相互关系，为后续功能分析和失效分析提供基础。

### DFMEA：结构树 + 方块图

**结构树示例（前照灯 LED 模组）**：

```
整车
 └ 系统：照明系统
    └ 子系统：前照灯
       └ 组件：LED 模组
          ├ 零件：LED 芯片
          ├ 零件：透镜
          ├ 零件：散热基板
          └ 零件：驱动 IC
```

**方块图（边界图）要素**：
- 关注项目（焦点要素）
- 上级要素（更高层级）
- 下级要素（更低层级）
- 接口要素（机械/电气/软件/信息）

### PFMEA：过程流程图 + 结构树 + 4M 工作要素

**过程流程图示例（注塑成型）**：

```
原料入库 → 原料干燥 → 注塑成型 → 冷却 → 脱模 → 去毛刺 → 外观检验 → 包装入库
```

**结构树示例**：

```
过程：注塑成型
 ├ 工序 1：原料干燥
 ├ 工序 2：注塑成型
 ├ 工序 3：去毛刺
 ├ 工序 4：外观检验
 └ 工序 5：包装
```

**4M 工作要素**（每个工序下展开）：

| 4M | 含义 | 示例 |
|---|---|---|
| Man（人） | 操作人员 | 操作工行为、技能、培训 |
| Machine（机） | 设备 | 注塑机、模具、机械手 |
| Material（料） | 材料 | PC 原料、色母、添加剂 |
| Method（法） | 方法 | SOP、参数设定、监控方法 |

**5M**：4M + Measurement（测）
**6M**：5M + Environment（环境）

### 输出物

- 结构树（Structure Tree）
- 方块图 / 边界图（DFMEA）
- 过程流程图（PFMEA）
- 4M 工作要素清单（PFMEA）

### 常见错误

- ❌ 结构树层级过深（超过 5 层）—— 难以分析
- ❌ 结构树层级过浅（只到组件层）—— 漏掉零件级失效
- ❌ PFMEA 未列出辅助工序（运输/储存）—— 漏分析
- ❌ 4M 分类错误（如把"参数设定"归到 Machine 而非 Method）

### IATF 16949 关联

- 条款 8.3.3.2（产品设计输入）
- 条款 8.5.1.2（标准化作业——操作员容易获得）

## 步骤三：功能分析

### 目的

明确每个系统要素/工序的功能及其要求，为失效分析（步骤四）提供基础。功能的完整定义会使失效分析更全面。

### DFMEA：功能树 + 参数图（P-图）

**功能定义规则**：
- 使用"主动动词 + 可测量名词"
- 用现在时态
- 关联到需求

**示例**：
- ✅ "将电能转换为光能"
- ✅ "提供 800 lm 光通量"
- ✅ "在 100 ms 内点亮"
- ❌ "照明"（动词太模糊）
- ❌ "好的前照灯"（无动词）

**功能树示例**：

```
前照灯 LED 模组：将电能转换为光能
 ├ LED 芯片：将电流转换为光辐射
 ├ 透镜：聚焦光束，形成指定光形
 ├ 散热基板：将 LED 热量传导至散热器
 └ 驱动 IC：恒流驱动 LED，提供调光控制
```

**参数图（P-图）要素**：

| 要素 | 说明 | 示例 |
|---|---|---|
| 输入（信号因素） | 系统接收的输入 | 电流、电压、控制信号 |
| 预期输出 | 系统期望输出 | 800 lm 光通量 |
| 非预期输出 | 系统不期望输出 | 热量、电磁辐射 |
| 控制因素 | 可调整的设计参数 | LED 选型、驱动电流、透镜设计 |
| 噪声因素 | 不可控的影响因素 | 见下表 |

**噪声因素 5 类**：

| 类型 | 示例 |
|---|---|
| 组件间变化 | LED 色温差异、透镜透光率差异 |
| 随时间变化 | LED 光衰、透镜黄变 |
| 顾客使用 | 频繁开关、超长时间开启 |
| 外部环境 | 高温、高湿、振动 |
| 系统交互 | 与其他灯具的 EMC 干扰 |

### PFMEA：过程功能分析

**工序功能定义**：
- 格式："工序名 + 作用对象 + 期望结果"
- 例："注塑成型工序：将 PC 原料转化为符合图纸的保险杠壳体"

**工作要素功能**：
- 设备功能：注塑机提供稳定的注射压力和保压
- 人员功能：操作工按 SOP 装卸模具
- 材料功能：PC 原料提供成型所需的物理特性
- 方法功能：参数设定规范保证成型质量

### 输出物

- 功能树 / 功能网（Function Tree / Network）
- 参数图（P-图，DFMEA）
- 功能要求清单
- 工序功能分析（PFMEA）

### 常见错误

- ❌ 功能描述模糊（"照明"而非"提供 800 lm 光通量"）
- ❌ 漏掉非预期输出（只写预期输出）
- ❌ 噪声因素分类错误（如把"客户使用"归到"外部环境"）
- ❌ PFMEA 未分析 4M 各自的功能

### IATF 16949 关联

- 条款 8.3.3.3（产品和过程的设计输入）
- 条款 8.3.5.2（制造过程设计输出）

## 步骤四：失效分析

### 目的

识别每个功能的潜在失效影响、失效模式和失效起因，并显示它们之间的关系（失效链），为风险分析提供基础。

### 失效链（Failure Chain）核心概念

```
失效影响（FE）── 失效模式（FM）── 失效起因（FC）
   ↑                ↑                ↑
为什么？         发生了什么？      为什么？
（后果）         （现象）         （根因）
```

- **FE（Failure Effect）**：失效对上级要素/最终用户/法规的影响
- **FM（Failure Mode）**：关注要素的功能失效方式
- **FC（Failure Cause）**：下级要素失效或噪声因素导致 FM

### 失效模式的 7 种类型

| 类型 | 说明 | 示例 |
|---|---|---|
| 功能丧失 | 无法操作、突然失效 | LED 不亮 |
| 功能退化 | 性能随时间损失 | LED 光通量从 800 lm 降到 400 lm |
| 功能间歇 | 随机开始/停止 | LED 偶发熄灭 |
| 部分功能丧失 | 性能损失 | LED 光通量从 800 lm 降到 600 lm |
| 非预期功能 | 错误时间/方向 | LED 在应熄灭时点亮 |
| 功能超范围 | 超出可接受极限 | LED 光通量超过 1000 lm（刺眼） |
| 功能延迟 | 非预期时间间隔后操作 | LED 在开关后 500ms 才点亮（应<100ms） |

### DFMEA 失效起因来源

1. **P-图噪声因素**（5 类）：
   - 组件间变化：LED 色温差异
   - 随时间变化：LED 光衰
   - 顾客使用：频繁开关导致驱动 IC 损坏
   - 外部环境：高温导致散热不足
   - 系统交互：EMC 干扰导致控制异常

2. **接口失效**：
   - 机械接口：连接器松动
   - 电气接口：电源电压异常
   - 软件接口：控制信号错误
   - 信息接口：通信协议错误

3. **设计缺陷**：
   - 材料选型：选用了不耐高温的材料
   - 几何设计：散热面积不足
   - 容差设计：公差过紧导致装配应力

### PFMEA 失效起因来源（4M1E 分类）

| 4M1E | 失效起因示例 |
|---|---|
| 人（Man） | 操作失误、培训不足、疲劳、未按 SOP |
| 机（Machine） | 设备故障、模具磨损、设备校准失效、参数漂移 |
| 料（Material） | 来料超差、批次差异、材料变更、运输损伤 |
| 法（Method） | SOP 缺失、参数错误、监控不足、防错失效 |
| 环（Environment） | 温湿度异常、洁净度不足、振动、电磁干扰 |
| 测（Measurement） | 量具误差、测量方法错误、测量系统失效 |

### 输出物

- 失效链分析（FE→FM→FC）
- 失效网（Failure Network）
- 失效分析表

### 常见错误

- ❌ 失效模式描述模糊（"质量不好"而非"LED 不亮"）
- ❌ 跳级（直接从 FM 到 FE，没有 FC）
- ❌ 一个 FM 只配一个 FC（应识别多个潜在 FC）
- ❌ PFMEA 的 FC 未按 4M1E 分类
- ❌ DFMEA 的 FC 未考虑 P-图噪声因素

### IATF 16949 关联

- 条款 8.3.5.2（制造过程设计输出）
- 条款 10.2.3（问题解决）

## 步骤五：风险分析

### 目的

通过评估严重度（S）、频度（O）、探测度（D）来估计风险，并对需要采取的措施进行优先排序。

### S 严重度（Severity）

**定义**：失效影响的严重程度，不考虑频度和探测度。

**评分表**（详见 `sod_scoring_tables.md`）：

| S | 等级 | 标准 |
|---|---|---|
| 10 | 非常高 | 影响车辆操作安全，危及驾驶员/乘客/行人健康 |
| 9 | 非常高 | 不符合法规 |
| 8 | 高 | 预期寿命内失去车辆主要功能 |
| 7 | 高 | 预期寿命内降低车辆主要功能 |
| 6 | 中 | 失去车辆次要功能 |
| 5 | 中 | 降低车辆次要功能 |
| 4 | 低 | 外观/声音/振动/触感非常不舒服 |
| 3 | 低 | 外观/声音/振动/触感中度不舒服 |
| 2 | 低 | 外观/声音/振动/触感略微不舒服 |
| 1 | 非常低 | 没有可觉察的影响 |

⚠️ **DFMEA 与 PFMEA 的 S 评分必须一致**（手册 1.4 节明确要求）

### O 频度（Occurrence）

**定义**：失效起因的发生频率，基于预防控制有效性。

**关键问题**（团队评级时参考）：

1. 类似组件/子系统/系统的过往使用情况和现场怎样？
2. 该项目是否是借用件或类似先前水平的项目？
3. 与先前水平的项目相比，变化有多大？
4. 该项目是完全新的吗？
5. 应用是什么或有哪些环境改变？
6. 是否使用了工程分析（如可靠性）来对预期频度等级进行估计？
7. 预防控制是否到位？
8. 在产品开发过程中，其稳健性是否得到验证？

**评分表**（详见 `sod_scoring_tables.md`）：

| O | 等级 | 关键特征 |
|---|---|---|
| 10 | 极高 | 新技术首次应用，无经验 |
| 7-9 | 高 | 创新设计/材料，预防控制有限 |
| 5-6 | 中 | 成熟技术，预防控制部分有效 |
| 3-4 | 低 | 类似设计，预防控制有效 |
| 1-2 | 非常低 | 长期验证，预防控制充分 |

### D 探测度（Detection）

**定义**：已发生的失效起因和/或失效模式的可探测程度。

**关键问题**：

1. 当前探测方法是什么？
2. 探测方法是否经过验证？
3. 探测时间点（在设计阶段/试产阶段/量产阶段）？
4. 探测能力（通过/不通过 vs. 失效测试 vs. 老化测试）？

**评分表**（详见 `sod_scoring_tables.md`）：

| D | 等级 | 关键特征 |
|---|---|---|
| 10 | 非常低 | 尚未制定测试过程 |
| 9 | 非常低 | 未为探测失效设计测试 |
| 7-8 | 低 | 新测试方法未验证/测试时间过迟 |
| 5-6 | 中 | 已验证测试，通过/不通过或失效测试 |
| 2-4 | 高 | 已验证测试，时间充分，提前发现 |
| 1 | 非常高 | 测试证明不会失效，或总是能探测到 |

### AP 行动优先级（Action Priority）

**2019 版核心变化**：不再使用 RPN（=S×O×D），改用 AP 三档优先级。

**AP 三档定义**：

| AP | 处置原则 |
|---|---|
| **H（高）** | 必须评审并采取措施，或证明并记录当前控制足够有效 |
| **M（中）** | 应评审并采取措施，或公司自行决定证明并记录当前控制足够有效 |
| **L（低）** | 可以确定措施来改进预防或探测控制 |

**AP 判定逻辑**：
1. **优先考虑 S**：S 越高，AP 越倾向于 H
2. **其次考虑 O**：O 越高，AP 越倾向于 H
3. **最后考虑 D**：D 越高，AP 越倾向于 H

**完整 1000 种组合**详见 `sod_scoring_tables.md`。

⚠️ 对于 S=9-10 且 AP=H/M 的失效影响，建议至少由管理层评审。

### 输出物

- S/O/D 评级表
- AP 优先级矩阵
- 风险分析表（含 PC/DC）
- 风险矩阵热力图（S×O、S×D）

### 常见错误

- ❌ 仍使用 RPN（如 RPN>100 必须改进）—— 2019 版已废弃
- ❌ S 评分主观臆断（不参考标准表）
- ❌ O 评分与预防控制脱节（未评估 PC 有效性）
- ❌ D 评分与探测控制脱节（未评估 DC 成熟度）
- ❌ DFMEA 与 PFMEA 的 S 评分不一致
- ❌ AP 判定错误（如 S=10, O=10, D=10 判为 L）

### IATF 16949 关联

- 条款 6.1（风险和机遇的应对措施）
- 条款 8.5.1.1（控制计划）

## 步骤六：优化

### 目的

确定降低风险的措施，并分配责任人和截止日期，跟踪措施实施和有效性。

### 措施分类

| 措施类型 | 作用 | 影响评分 |
|---|---|---|
| 预防控制（PC）改进 | 降低失效起因发生概率 | 降低 O |
| 探测控制（DC）改进 | 提升失效探测能力 | 降低 D |
| 设计/过程变更 | 降低失效影响严重度 | 降低 S（通常较难） |

### PC 改进措施示例

- 增加设计规范、设计指南
- 增加 DOE（试验设计）优化参数
- 增加仿真分析（CAE、CFD、FEA）
- 增加可靠性试验（HALT、HASS）
- 增加供应商管理（来料控制）
- 增加防错设计（Poka-Yoke）
- 增加冗余设计

### DC 改进措施示例

- 增加测试项目（功能测试、性能测试、可靠性测试）
- 提前测试时间（在设计阶段而非量产阶段）
- 采用更敏感的测试方法（如失效测试代替通过/不通过）
- 增加老化测试
- 增加在线自动检测
- 增加 SPC 监控

### 措施状态管理

| 状态 | 说明 |
|---|---|
| 已建议（Suggested） | 措施已提出，待评审 |
| 已决策（Decided） | 措施已评审通过，含责任人和截止日期 |
| 已实施（Implemented） | 措施已实施，含实施日期 |
| 已验证（Verified） | 措施有效性已验证，S/O/D 重新评级 |
| 已关闭（Closed） | 措施完成，AP 已降低，关闭人签字 |

### 措施有效性评估

措施实施后，重新评估 S/O/D：
- 如果 S/O/D 降低 → AP 重新判定
- 如果 AP 降到 L → 措施可关闭
- 如果 AP 仍为 H/M → 需要进一步措施或管理层决策

### 输出物

- 措施清单（含责任人、截止日期、状态）
- 措施有效性评估记录
- 措施关闭记录

### 常见错误

- ❌ 措施无责任人、无截止日期
- ❌ 措施实施后未重新评估 S/O/D
- ❌ S=9-10 的措施未由管理层评审
- ❌ 措施只改进 DC 而不改进 PC（应优先改进 PC）

### IATF 16949 关联

- 条款 10.2.3（问题解决）
- 条款 9.1.1.1（监视和测量资源）

包含 7 个章节：

2. **结构分析**：结构树 + 接口
3. **功能分析**：功能树 + 参数图
4. **失效分析**：失效链清单
5. **风险分析**：S/O/D 评级 + AP 矩阵
6. **优化措施**：措施清单 + 责任人
7. **结论与建议**：高风险项汇总、特殊特性清单

### 特殊特性识别

- **CC（Critical Characteristic，关键特性）**：S=9-10 的失效影响对应的特性
- **SC（Special Characteristic，特殊特性）**：S=8 且 AP=H/M 的失效影响对应的特性
- **OSS（Operator Safety Characteristic，操作员安全特性）**：影响操作人员安全的特性

识别出的 CC/SC 应同步到控制计划（CP）中。

### 输出物

- FMEA 表格（.xlsx，7 Sheet）
- FMEA 报告（.docx，7 章）
- 特殊特性清单（CC/SC/OSS）
- 控制计划草稿（可选）

### 常见错误

- ❌ 表格不完整（缺 Sheet 或字段）
- ❌ 措施状态未更新（仍是"已建议"）
- ❌ CC/SC 未识别或未同步到控制计划
- ❌ 报告未签字/未评审

### IATF 16949 关联

- 条款 8.3.5.2（制造过程设计输出）
- 条款 8.5.1.1（控制计划）

## 与其他工具的关联

### 与 8D 报告的关联

- FMEA 是预防性分析，8D 是事后问题解决
- 8D 的 D4 根因分析可能发现 FMEA 未覆盖的失效模式 → 应回写到 FMEA
- 8D 的 D7 横向展开可能触发 FMEA 修订

### 与 APQP 的关联

- FMEA 在 APQP 的第 2 阶段（产品设计和开发）与第 3 阶段（过程设计和开发）启动
- FMEA 输出是控制计划（CP）的输入
- FMEA 在 PPAP 提交时作为必备文件

### 与控制计划（CP）的关联

- FMEA 识别的 CC/SC → CP 的特殊特性
- FMEA 的 PC/DC → CP 的控制方法
- FMEA 的 AP=H 项 → CP 的重点控制项
